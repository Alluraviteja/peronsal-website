🤖 Rate Limiter Agents

A monitoring layer for rate-limit telemetry that combines deterministic rule evaluation with constrained LLM-based analysis. Three specialist agents analyze errors, token health, and traffic patterns on a scheduled interval; an orchestrator synthesises their signals into a recommended action monitor, alert, throttle, or block. All outputs are signals only; no automated infrastructure changes are triggered.

TL;DR

  • 3 specialist agents analyze errors, token health, and endpoint traffic independently each writes its result before the next one starts
  • A rule engine applies deterministic severity escalation; the LLM performs contextual interpretation on top
  • Pipeline runs per app on a configurable interval every result tagged with app_info_id for full isolation
  • Two-layer memory: 7-day rolling baseline + last 3 run results injected into the orchestrator prompt
  • 5-scenario daily regression suite catches prompt drift before it reaches production
  • Strategy + Factory pattern for LLM backends swap providers by changing one environment variable

Most rate limiters expose raw telemetry but little contextual interpretation. I built this project to explore whether specialist LLM agents could act as a lightweight contextual analysis layer on top of deterministic infrastructure and to test where that boundary between rules and interpretation should sit. I intentionally split responsibilities into specialist agents to keep prompts narrow, reduce cross-signal ambiguity, and make failures easier to isolate during evaluation.

Architecture

How It's Wired Together

The orchestrator receives all three specialist results as in-memory Python objects it never re-queries the agent database for the current run's data. This avoids redundant reads and eliminates consistency issues between agent writes and orchestrator reads within the same pipeline execution.

Data layer & startup details

Two separate databases with distinct roles. The rate limiter database is the read model raw telemetry, never written to, no migrations run against it. The agent database is the write model results, baseline memory, eval outcomes. Alembic migrations run on startup and apply only to the write side.

On startup the application runs Alembic migrations to head before starting the APScheduler background scheduler. If migrations fail, startup aborts no half-initialised scheduler running against a stale schema. Both SQLAlchemy engines carry pool_pre_ping=True, which validates connection liveness before every query and reconnects transparently if the database restarted.

Each pipeline run follows a fixed lifecycle: fetch all enabled apps → for each app, run agents ①②③ sequentially → run orchestrator ④ → log the final verdict. Each specialist agent writes its own result before the next starts.

Overview

What It Does

APScheduler triggers a pipeline run on a configurable interval (default 15 minutes). Each run fetches all enabled apps and executes the four-agent pipeline once per app every result tagged with app_info_id for full tenant isolation. Up to 5,000 log rows per agent per run keeps query cost predictable regardless of traffic volume. 4 LLM calls per app, every result persisted with severity, action, reason, token usage, and USD cost.

Each app carries a per_ip_address flag that changes how every agent interprets the same numbers. A 90% block rate on a per-IP app where one client is the culprit is normal operation; the same number on a shared-bucket app affects every caller. See Behavioral Flag Pattern below for how this is enforced across all agents.

Agents

The Four Agents

Specialist + Orchestrator pattern. Each specialist owns a single signal, executes as a stateless agent, and writes its result before the next one starts. The orchestrator is the first component with access to all three outputs.

Sequential execution was intentionally chosen for the initial design because deterministic ordering simplified debugging, replay analysis, and regression evaluation during prompt iteration.

🚨

① Error Pattern Agent

Block rate, error rate, and response code distribution over the last 15 minutes. Surfaces per-path error concentrations and top block reasons.

  • Anomaly when error_rate > 20%, block_rate > 30%, or any 5xx response

④ Orchestrator

Two-phase synthesis: a deterministic rule engine sets the severity floor, then a context-augmented LLM confirms or adjusts using baseline and recent history.

Phase 1 Deterministic Escalation

  • Any agent at critical → final severity is critical
  • 2+ agents at high → critical · 1 agent at high → high
  • 2+ agents at medium → high · otherwise inherit the highest

Phase 2 Contextual Validation

  • 7-day rolling baseline + last 3 orchestrator results injected into prompt
  • Escalating trend → upgrade one level · stable or recovering → downgrade

Example

Incident Walkthrough

A shared-bucket app starts showing unusual patterns simultaneously across three signals during a 15-minute window.

Observed Signals

  • 82% block rate, 31% error rate (no 5xx)
  • 65% of requests have near-zero tokens remaining
  • /login consuming 91% of total traffic
  • Severity escalating across the last 3 recorded runs

Agent Outputs

  • Error Agent → high (block rate > 30%, error rate > 30%)
  • Token Agent → critical (near-depletion > 70%)
  • Paths Agent → high (/login at 91% traffic concentration)

Orchestrator Decision

Phase 1 (rule engine): 1 critical agent → deterministic floor is critical. Mapped action: block (signal only).

Phase 2 (LLM context): 3-run trend shows escalating severity. Baseline block rate for this app is 12% current 82% is a 6.8× spike. LLM confirms: no downgrade warranted.

Raw LLM output & parse

The LLM prompt specifies an exact labeled format and forbids free-form text. The response is parsed line-by-line no unstructured text reaches the database.

ANOMALY: YES
ACTION: block
REASON: Critical token depletion with concentrated /login traffic and escalating block rate across three consecutive runs.

Implementation

Key Design Decisions

★ Core

Constrained Generation + Null Object Fallback

Agents use constrained generation the LLM prompt specifies an exact labeled format and forbids free-form text. The response is parsed line by line into key-value pairs.

  • Missing or unparseable field → safe default via .get("KEY", default)
  • Full agent exception → Null Object: severity none, action monitor, cost $0
  • One bad LLM response never halts the rest of the pipeline run
★ Core

Two-Layer Agent Memory

Long-term memory is a sliding window aggregate a running mean of RPS, block rate, and bot ratio per app, capped at 672 samples (~7 days at 15-min intervals).

  • Spike threshold derived dynamically: 2.5× if baseline block < 5%, 3× normal, 4× if > 20%
  • Short-term: last 3 orchestrator results serialised into the prompt as recent history
  • Baseline is O(1) per app a single row stores a running mean capped at 672 samples; it never grows. Historical incident rows (AgentResult, OrchestratorResult) accumulate separately as an unbounded audit trail
★ Core

Behavioral Flag Pattern

The per_ip_address flag is read at the start of each pipeline run and passed explicitly into every agent call and LLM prompt. Same raw metrics, completely different meaning depending on mode.

  • Flag is stated explicitly in every prompt the LLM is never asked to infer context
  • Per-IP severity downgrade logic is encoded directly in each agent's system prompt
4 more design decisions

Strategy + Factory for LLM Backends

A shared abstract base class defines the contract complete() returns a normalised LLMResponse with content, token counts, and USD cost.

  • Factory reads LLM_PROVIDER env var at startup no conditionals in agent code
  • Adding a new provider means one new class; nothing else changes
  • Retry with exponential backoff lives in the base class 3 attempts, 2 s base delay with jitter

Scenario-Based Regression Testing

Each eval scenario uses a test double a fresh in-memory SQLite database seeded with synthetic logs for a known traffic condition.

  • 5 scenarios × 4 agents = 20 severity and action accuracy checks per daily eval run
  • System prompts are versioned with the codebase git history links any accuracy drop to the exact prompt change that caused it
  • Results stored as a continuous accuracy trend queryable at /evals; degradation is visible before it affects production

Cost Attribution

USD cost is calculated at write time and stored per agent result not as a single pipeline total.

  • Attributed per agent name, per app, and per day
  • Dashboard exposes a 7-day cost series broken down by agent
  • Makes optimisation decisions concrete: which agent costs most relative to its signal weight?

Immutable Artifacts & Automated Rollback

Every image is tagged with the exact commit SHA. CI gates every push; CD only runs after CI passes.

  • CI: TruffleHog secrets scan, mypy, ruff, pytest, Trivy filesystem + image scan
  • CD polls /health (liveness) every 5 s for 90 s via Docker HEALTHCHECK on failure, restarts previous SHA-tagged image
  • Multi-stage Docker: compiler toolchain only in build stage; tini as PID 1, non-root user

Design Rationale

Why LLMs?

The rule engine handles severity escalation. The LLMs handle the parts where a pure rule system would need a combinatorial matrix that's hard to maintain and harder to reason about.

Comparison of where a rules engine excels versus where the LLM layer adds value
Problem Rules Engine LLM Layer
Hard thresholds & severity escalation Excellent fast, deterministic Unnecessary overhead
Trend interpretation (escalating / recovering) Complex stateful logic Natural reads serialised history
Human-readable incident summaries Difficult static templates only Excellent contextual prose
Cross-signal reasoning (mode × severity × signal) Rule explosion risk Flexible handles combinations naturally
Deterministic guarantees Excellent Weak non-deterministic at margins
  • Cross-signal interpretation and context-dependent thresholds block rate, error rate, 5xx presence, and per-IP mode interact combinatorially. A single LLM prompt that understands mode is simpler than a rule tree that branches on mode × severity × signal combination.
  • Trend analysis the orchestrator compares recent structured verdict history against the rolling baseline to detect escalation or recovery patterns.
  • Incident summarisation every verdict includes a human-readable reason field that explains the decision in context. This cannot come from a rule engine.

The boundary is equally important. LLMs are never permitted to:

  • Trigger infrastructure actions the block/throttle/alert verdict is a signal only; no automated changes to rate limits or traffic policies are ever triggered by LLM output
  • Bypass deterministic escalation the rule engine runs before every LLM call; LLM output cannot override the severity floor
  • Write free-form state LLM output is parsed into structured fields (SEVERITY, ACTION, REASON); no unstructured text reaches the database
  • Mutate telemetry the rate limiter DB is strictly read-only for this service

Scale & Resilience

Tradeoffs & Failure Modes

Pipeline time scales as O(apps × 4 LLM calls × latency). With a small number of monitored apps this is fine; with many, LLM latency becomes the bottleneck.

2–5 s per pipeline run (single app)
300–800 ms avg LLM call latency
~$0.004–0.012 per full pipeline run (4 calls)
O(1) rolling baseline memory (constant regardless of runtime)
~$8–15/day 15-min interval across 10–20 apps (96 runs/day/app)
  • Sequential agents & synchronous I/O sequential execution was the intentional choice for the initial design: deterministic ordering simplified observability, replay analysis, and failure isolation during prompt iteration. LLM calls are currently synchronous (blocking retry with thread-sleep), DB queries use standard SQLAlchemy sessions, and APScheduler dispatches pipeline jobs on a thread-pool executor. This makes each run fully traceable but means latency scales as O(apps × 4 calls × LLM latency). At larger scale, the pipeline would shift to per-app async concurrency via asyncio.gather with a bounded semaphore controlling LLM fanout parallelising across apps while keeping within-app sequential order and avoiding provider rate-limit bursts.
  • 3 independent DB queries per window each specialist agent re-queries the same log window separately. Simpler isolation, but 3× duplicate reads versus a shared fetch model.
  • In-process scheduler APScheduler runs in the same process as FastAPI. A scheduler hang affects the API and vice versa. A separate worker process would isolate these concerns.
  • LLM non-determinism identical inputs can produce slightly different severity outputs across runs. The deterministic rule engine sets a hard floor, but exact reproducibility is impossible at the margins.
Each failure mode and how it is handled by the system
Failure Mode Mitigation
Malformed LLM output Constrained generation + Null Object fallback missing fields default to none/monitor, never raise an exception
Provider timeout or rate limit Exponential backoff retry 3 attempts, 2 s base delay with jitter, then safe default result
Database unavailable pool_pre_ping=True on both engines; /health/ready returns 503 if either DB is down
Prompt or model regression Daily eval suite re-tests 5 labelled scenarios and stores an accuracy trend drift visible before it affects production
Agent stall / hung provider call Per-agent execution timeout prevents a stalled provider call from blocking the entire pipeline run indefinitely. Retries count toward the total budget so one degraded provider cannot stall the pipeline.
Container crash on deploy CD polls /health (liveness, via Docker HEALTHCHECK) for 90 s; on failure, restarts the previous SHA-tagged image automatically

Deep Dives

More Detail

Observability

Structured logs, readiness/liveness probes, and dashboard APIs expose runtime state with no separate monitoring stack required.

Structured JSON Logs

A custom formatter wraps every log entry: timestamp, level, logger, message, and optionally a request_id.

  • X-Request-ID echoed on every HTTP response correlate a request across log lines by a single ID
  • Per-app pipeline completion: app_info_id, severity, action on every run
  • Per-app pipeline failures logged individually one app crashing does not suppress other apps' logs

Health Endpoints

Two tiers of health signal, aligned with standard container orchestration patterns.

  • /health liveness: returns {"status": "ok"} if the process is up
  • /health/ready readiness: runs SELECT 1 against both DBs; returns 503 if either is unreachable
  • 503 on readiness stops traffic routing and triggers automatic rollback to the previous image

Dashboard API

Three REST endpoints expose runtime state without a separate observability stack.

  • /dashboard 7-day LLM cost series broken down by agent name and day
  • /agents recent agent results per app: severity, action, reason, token usage
  • /evals daily accuracy trend: severity_accuracy_pct and action_accuracy_pct per run
