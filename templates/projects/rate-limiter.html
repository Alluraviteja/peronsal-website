⏱️ Rate Limiter

A standalone rate limiting service built with Spring Boot. Rather than baking limits into every API, this runs as a shared component — any upstream gateway can call it, get a yes/no back, and forward the result to the client as standard rate-limit headers.

Spring Boot Token Bucket Fail-open / Fail-closed Redis Bucket4j Lettuce Lua Script PostgreSQL Docker Resilience4j CI/CD

Overview

What It Does

You register an app, assign it a rate limit plan, and this service does the rest. When a request comes in, the caller sends a service identifier and client IP; the service checks the token bucket in Redis and responds with allowed, remaining tokens, and retry-after. The caller can pass those straight back as RateLimit-* headers — no extra work on their end.

Every decision — allowed or blocked — gets written to PostgreSQL with the full request context: IP, path, HTTP method, device type, browser, OS, and whether it looked like a bot. That audit log makes it straightforward to investigate abuse or tune limits after the fact.

Algorithm

Rate Limiting Strategy

Token Bucket via Bucket4j, with all state stored in Redis. Because the bucket lives in Redis rather than in-process, the limit holds the same whether you're running one instance or ten.

🪣

Token Bucket (Bucket4j + Redis)

Each app gets a bucket with a capacity ceiling and a steady refill rate. Incoming requests spend tokens; once the bucket is empty, requests are blocked until tokens refill.

  • Handles short bursts — callers can spend up to the full capacity before hitting the limit
  • Greedy refill — tokens trickle back in continuously rather than all at once
  • Bucket state lives in Redis — consistent across every running instance
  • Read-modify-write is a single atomic Lua script — no race conditions under concurrency

Architecture

How It's Wired Together

Kept the data model intentionally split: Redis holds live token counts with a TTL matching each plan's refill period — nothing lingers longer than it needs to. PostgreSQL holds the durable stuff: app registrations, rate limit plans, and the full audit log. Spring Boot wires the two together and exposes the HTTP API.

Implementation

Key Design Decisions

Atomic Lua Scripts

The problem with distributed rate limiting is the read-modify-write gap. If two nodes both read "one token left" before either writes back, both requests slip through. Bucket4j solves this by sending the entire check — read, refill, decrement, write — as a single Lua script. Redis executes Lua atomically, so no two requests can interleave on the same key.

Lettuce Async Client

Lettuce multiplexes many concurrent requests over one connection, so Redis calls don't block threads while waiting on the network. Under load, that adds up — one shared connection handles everything without the pool pressure you'd see with a blocking client.

Config-Driven Plans

Each app's capacity, refill rate, and refill window are stored in PostgreSQL and loaded lazily on first use, then held in an in-process cache. Updating a plan in the database takes effect on the next request with no restart needed. The cache can also be flushed manually via an admin endpoint.

Per-IP or Per-App Limiting

When registering an app you choose how quota is scoped. Per-app mode gives the whole service a single shared bucket — good for capping total throughput. Per-IP mode gives each caller their own — better for public-facing APIs where you don't want one client exhausting everyone else's quota. The bucket key in Redis changes based on that flag.

Circuit Breaker & Retry

Redis calls go through Resilience4j — a retry layer for transient blips (two attempts, 50ms apart, only on Redis exceptions) and a count-based circuit breaker that trips when 50% of calls fail over a 20-request window. Once open, it waits 10 seconds before testing again. The goal is to stop hammering a Redis instance that's clearly down and recover automatically when it comes back.

Fail-Open / Fail-Closed

If Redis is unreachable and retries are exhausted, each app has a flag controlling what happens next. Fail-open lets the request through — availability matters more than enforcement. Fail-closed returns a 503 — correctness matters more. Both outcomes are logged so you can see how often the fallback is actually firing in production.
