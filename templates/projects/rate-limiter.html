⏱️ Rate Limiter

A distributed, high-performance rate limiting service built with Spring Boot, using the Token Bucket algorithm backed by Redis to protect APIs from abuse and ensure fair usage across clients.

Spring Boot Maven Bucket4j Lettuce Lua Script PostgreSQL Redis

Overview

What It Does

The Rate Limiter acts as a middleware layer that controls the rate of incoming API requests on a per-client basis. Rather than embedding rate limiting logic into individual services, this runs as a standalone service that any upstream gateway or service can consult before processing a request.

It exposes a simple HTTP API: given a client identifier and an endpoint key, it returns allowed, remaining tokens, and retry-after — so callers can forward the request or return a meaningful 429 with standard rate-limit headers.

Algorithm

Rate Limiting Strategy

Token Bucket via Bucket4j, backed by Redis. All bucket state lives in Redis so the limit is enforced consistently across every instance of the service.

🪣

Token Bucket (Bucket4j + Redis)

Each registered app gets a bucket with a fixed capacity that refills at a constant rate. Requests consume tokens; if the bucket is empty, the request is throttled.

  • Allows short bursts up to bucket capacity
  • Smooth greedy refill via configurable rate and period
  • Distributed — bucket state stored in Redis, shared across all instances
  • Atomic read-modify-write via Bucket4j Lua script — no race conditions

Architecture

How It's Wired Together

The service is intentionally thin. Spring Boot handles the API surface and dependency wiring. Redis via Lettuce holds transient rate state, and PostgreSQL persists client configurations and audit records.

Implementation

Key Design Decisions

Atomic Lua Scripts

Every token-bucket check is a single Lua script executed atomically on Redis: GET the serialized bucket state, apply refill logic, check availability, SET the updated state back with a TTL — all in one operation with no interleaving.

Lettuce Async Client

Lettuce's async commands keep Redis I/O non-blocking. A single thread-safe, multiplexed connection is shared across all concurrent requests — no pool thrash, no stalling on network round-trips.

Config-Driven Rules

Rate limit rules — capacity, refill rate, and refill period — are stored in PostgreSQL per app and lazy-loaded on first request, then cached in-process. Rule updates take effect without a redeployment.

Per-IP or Per-App Limiting

Each registered app can be configured to enforce limits per IP address or per app as a whole. The bucket key is scoped accordingly, giving fine-grained control over how quota is shared across callers.

Async Audit Trail

Every rate decision is handed off to an async audit queue — fire-and-forget so the hot path stays fast. Denied requests and a sample of allowed ones are written to PostgreSQL (the same instance as the config store) for analytics and abuse investigation.

Failure Handling

If Redis is unreachable — timeout or network error — the service can be configured to fail-open (allow all traffic, prioritize availability) or fail-closed (block all traffic, prioritize correctness). The right choice depends on the caller's risk tolerance.
